# Things that DON'T work (so you don't repeat them)

Every row below was tried on this exact box and rejected, with the number that killed it. Keep this list handy before you "improve" the config — most of these look like obvious wins and cost a day each to disprove. (Benchmark-only rejects with their throughput deltas also live in [`../bench/RESULTS.md`](../bench/RESULTS.md#rejected-with-numbers-so-nobody-redoes-them).)

> **The `turboquant_4bit_nc` arc — un-rejected 2026-07-15, re-rejected for good 2026-07-20.** It used to head this list ("4-bit keys destroy retrieval, 0/8"). That 0/8 was a **confound** — vLLM's async×spec-decode scheduler desync ([#42655](https://github.com/vllm-project/vllm/issues/42655)), not the 4-bit keys — and with **`--no-async-scheduling`** it passed everything (8/8 all depths, 90/90 concurrent) and ran as the daily for three days ([the reversal](HISTORY.md#status-turboquant_4bit_nc-is-the-daily-the-asyncspec-reversal)). It was then superseded on merit by fp8 KV (flat deep decode, no custom kernel), and a final re-audition on the modern tier stack closed it permanently: 563K-token pool, wrong 60K needle, dead under the concurrency killer ([the final word](HISTORY.md#status-the-daily-added-lmcache-dramnvme-kv-tiers-2026-07-20)). Both lessons stand: suspect confounds behind every *"corrupts under concurrency"* verdict below — and a 2.6× pool that fails retrieval is worth nothing.

| tried | verdict |
|---|---|
| **nvfp4-FA2** (FlashInfer FA2 nvfp4 KV) | **Works, but loses to `turboquant_4bit_nc` on every axis — and was far more work.** We built the "proper" nvfp4 KV path — [jethac/vllm](https://github.com/jethac/vllm) branch + [FlashInfer PR #3684](https://github.com/flashinfer-ai/flashinfer/pull/3684), JIT-compiled sm120 kernels — the same nvfp4 TechPrototyper documented at a 416K pool. It's byte-identical, retrieval is clean, and it runs on the FlashInfer FA2 backend. But: **stable pool only 184K** (+11% vs k8v4, weights-bound by *our* vision+MTP checkpoint — the 416K needs a lighter text-only model), **decode −8 to −23%** (worst at deep single-stream, from FlashInfer disabling split-KV on 4-bit KV), **tool-eval 82**, **OOMs at util 0.97** (needs 0.94), and it's a **2-unmerged-branch dev build** with a ~15-min first-launch JIT compile. Punchline: we built nvfp4-FA2 the hard way, then found **`turboquant_4bit_nc` — same idea, better result — was a preset flag on the image we already had.** (`4bit_nc` has itself since been retired — see its row — which only strengthens this reject: the surviving fp8 daily needs no dev build at all.) |
| **smaller-bit TurboQuant presets** (`k3v4_nc`, `3bit_nc`) | **Dead — the code's own PPL table walls them off.** Perplexity delta vs bf16: **k8v4 +1.17% → 4bit_nc +2.71% → k3v4_nc +10.63% → 3bit_nc +20.59%.** Every preset smaller than `4bit_nc` attacks the **keys** (3-bit keys), and keys are the sweet spot — retrieval collapses. There is **no** 8-bit-key/smaller-value preset to try (values are already at 4). `4bit_nc` is the smallest preset that keeps quality; below it is a cliff. |
| **`turboquant_4bit_nc`** | **Re-rejected — closed permanently (see the arc above).** Un-rejected once the async×spec confound was removed (8/8 needles, 90/90 concurrent, ~235K pool; the daily 2026-07-15 → 18), then superseded by fp8 KV (flat deep decode, battle-tested, no custom kernel), and finally re-auditioned on the modern tier stack: **pool 563,888 — but a wrong 60K needle (hybrid buffer co-location corruption) and a dead engine under the `pp8192×c8` killer.** Revival needs a state-corruption fix + a concurrency fix + a deep-decode-crater fix + new LMCache serde for its packed layout, against a working fp8 tier stack. Not worth it. |
| **KVarN** (vLLM fork, 4-bit KV quant) | **Disqualified under concurrency — the same failure class as TurboQuant.** As the big-context alternate it passed every single-stream depth/tool test, then failed both ways under concurrent load: with MTP, a **fatal CUDA device assert**; without spec decode, **`!!!!` garbage at 3 × 45K streams**. Honest caveat: this predates the [#42655](https://github.com/vllm-project/vllm/issues/42655) async-confound discovery and KVarN was never re-run with `--no-async-scheduling` — but it shares the architecture the TurboQuant post-mortem identified (custom 4-bit forks bypass native varlen attention and rebuild block tables under batch-uniformity assumptions), and TurboQuant's own async-off re-audition still failed. Across every concurrency battery on this hybrid, only fp8 KV has stayed clean. |
| **`nvfp4` native KV cache** | **Won't load.** `head_size=256` has no FlashInfer nvfp4 backend on `sm_120` via the *stock* path — needs unmerged [vLLM PR #44389](https://github.com/vllm-project/vllm/pull/44389). (The **nvfp4-FA2** row above is the branch-build that *does* load it — and still loses to `4bit_nc`.) |
| **DFlash** speculative decoding | **Works, but boxed in — measured and rejected.** Requires a full source build of [PR #40898](https://github.com/vllm-project/vllm/pull/40898) (SWA draft support). Real result with the [z-lab draft](https://huggingface.co/z-lab/Qwen3.6-27B-DFlash): **185 t/s single-stream (2.0× its no-spec baseline — a bigger uplift than MTP's ~1.6×)**… at the cost of **21K max context** (3.3 GB draft + bf16-KV-only: fp8 trips the branch's hybrid page-size assert, and the branch predates NVFP4-`lm_head` support so quantized targets are limited), **zero batch scaling** (c4 aggregate ≈ c1), and ~3× slower prefill. MTP's draft head lives *inside* the weights, costs ~0, and keeps 200K+ context. Revisit if #40898 merges into nightly. |
| `nvidia/Qwen3.6-27B-NVFP4` (official) | ~2.6× slower prefill, 20–25% slower decode, fatter checkpoint (max ~150K ctx), MTP crashes at moderate batch. Community quants win. |
| **`--async-scheduling`** (i.e. *not* passing `--no-async-scheduling`) | **Rejected twice over.** Originally on throughput (c4 552 → 526). But the bigger reason: with MTP it **desyncs the request-ID→batch-row mapping and corrupts KV** ([#42655](https://github.com/vllm-project/vllm/issues/42655)) — 0/8 retrieval on `4bit_nc`, ~10% intermittent on k8v4. `--no-async-scheduling` has been **mandatory** on every MTP daily since — TurboQuant then, fp8 now. |
| [froggeric fixed chat templates](https://huggingface.co/froggeric/Qwen-Fixed-Chat-Templates) | Bundled template already scores **4/4** on a behavioural probe (single + parallel tool calls, chat→tool→chat, chat-after-tools). Zero measured gain. |
| `--max-num-batched-tokens 4096` | **~4× prefill regression** (9.6K → 2.6K t/s) for +28K ctx. Keep 8192. |
| `VLLM_TQ_KV_SPLITS` < 32 | Hurts *both* c1 (139→132) and c8. Not the batching bottleneck. |
| `VLLM_NVFP4_GEMM_BACKEND=flashinfer-cutlass` | Already the nightly default. No-op. |
| **LMCache + TurboQuant KV** (`turboquant_k8v4`, persisted tier) | **Composes, but the persisted L2 tier is lossy — not shipped.** k8v4 KV builds and runs under LMCache and stores land, but the L2 SSD reload silently drops long-context needles (7/7 miss) — the format-10 kernel copies the standard 512-wide layout, not k8v4's 262-wide packed one. (`turboquant_4bit_nc` packs even tighter, so it would have been lossier still.) LMCache persistence only round-trips faithfully with fp8 KV. Full write-up: [`LMCACHE.md`](LMCACHE.md#lmcache--k8v4-composes-but-the-persisted-tier-is-lossy--not-shipped). |
